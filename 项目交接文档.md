## 六、FIT image 机制（必须理解）

Luckfox 250802 的 `boot.img` 是 **FIT image（Flattened Image Tree）** 格式，不是普通 Rockchip 分区格式。

```
offset 0x000:  FIT header DTB（1536 bytes）— 存各组件 SHA256 hash
offset 0x800:  FDT blob（原始 39020 bytes）— 设备树
offset 0xa200: kernel（~3.2MB）
offset 0x321200: resource.img
offset 0x321a00: rk-kernel.dtb 副本（39020 bytes）
```

**核心陷阱**：直接替换 DTB 字节不生效。U-Boot 验证 FIT header 中的 SHA256 hash，不匹配则回退到旧 DTB。必须同时更新 hash。Python 更新脚本在 `GC2093完整配置指南.md` 第 6.5 节。

当新 DTB > 原始 DTB 大小时，需改用 `mkimage` 重建 FIT（见 `常见问题FAQ.md` Q16）。

## 七、GC2093 DTS 核心配置

```dts
/* rv1106-luckfox-pico-zero-ipc.dtsi */
gc2093: gc2093@37 {
    compatible = "galaxycore,gc2093";
    status = "okay";
    reg = <0x37>;
    clocks = <&out_osc_cam>;
    clock-names = "xvclk";
    reset-gpios = <&gpio3 RK_PC5 GPIO_ACTIVE_LOW>;
    pinctrl-0 = <&cam_io_1>;
    rockchip,camera-module-index = <0>;
    rockchip,camera-module-facing = "back";
    rockchip,camera-module-name = "LUCKFOX-GC2093";    /* IQ 文件名依赖此值 */
    rockchip,camera-module-lens-name = "30IRC-F20";    /* IQ 文件名依赖此值 */
    port {
        gc2093_out: endpoint {
            remote-endpoint = <&csi_dphy_input1>;
            data-lanes = <1 2>;  /* 2-lane MIPI */
        };
    };
};
```

IQ 文件命名规则：`{sensor}_{module-name}_{lens-name}.json`  
→ `gc2093_LUCKFOX-GC2093_30IRC-F20.json`

## 九、SDK 限制（重要）

Luckfox Pico SDK 源码**不完整**（约 166 个内核源文件缺失），只能：
- ✅ 编译 DTB
- ✅ 编译单独驱动模块（gc2093.ko）
- ❌ 无法编译完整 kernel / 完整固件

在 macOS 上用 Docker 编译 DTB 时加 `--platform linux/amd64`（Apple Silicon 需要 Rosetta 2 模拟）。

```bash
SDK_DIR="/Users/lss/Desktop/瑞芯微/rv1106/luckfox-pico-sdk"
docker run --rm --platform linux/amd64 \
  -v "$SDK_DIR:/sdk" ubuntu:22.04 bash -c "
    apt-get install -y -qq make gcc bc bison flex libssl-dev \
      device-tree-compiler python3 python-is-python3 > /dev/null 2>&1
    cd /sdk/sysdrv/source/kernel
    make ARCH=arm luckfox_rv1106_linux_defconfig
    make ARCH=arm rv1106g-luckfox-pico-zero.dtb -j\$(nproc)
    cp arch/arm/boot/dts/rv1106g-luckfox-pico-zero.dtb /sdk/output_dtb.dtb
  "
```

## 十一、已知陷阱速查

| 陷阱 | 原因 | 解决 |
|------|------|------|
| DTB 改了但不生效 | FIT hash 未更新 | 用 Python 脚本更新 hash，或用 mkimage 重建 |
| rkipc 找不到 IQ 文件 | 文件名必须精确匹配 DTS 模组名 | `cp gc2093_SIDA... gc2093_LUCKFOX-GC2093_30IRC-F20.json` |
| rkipc 启动后崩溃 | GC2093 JPEG 编码器 timeout | ini 中 `enable_jpeg = 0` |
| RTSP 连不上 | macOS VPN 劫持 172.32.x.x 路由 | `adb forward tcp:15554 tcp:554` 用 localhost |
| rkdeveloptool db 挂起 | macOS maskrom USB 兼容性 | 改用 `adb reboot loader` 进 loader 模式 |
| 内核编译报 xt_HL.o | macOS HFS+ 大小写不敏感 | 禁用相关 Kconfig 项（见 SDK编译指南.md 第 11.3 节） |
| insmod 报 invalid module format | vermagic 不匹配 | 必须用 `luckfox_rv1106_linux_defconfig` 编译 |
| adb shell 后台进程被 kill | session 关闭时进程随之终止 | 用 nohup 或 RkLunch.sh 开机自启 |

## 十一、GC2093 SDK 固化配置（2026-07-10）

已将 Pico Zero 的 GC2093 适配写入 SDK 的默认 IPC 板级配置。以后基于同一套
`luckfox-pico-sdk` 构建 Pico Zero 固件时，不需要再手工推送驱动、复制 IQ 或替换
RKIPC 配置：

- `BoardConfig-EMMC-Buildroot-RV1106_Luckfox_Pico_Zero-IPC.mk` 默认打包 GC2093 原始 IQ；
- `rv1106-luckfox-pico-zero-ipc.dtsi` 固化 37.125MHz 时钟、GC2093/MIPI 端点和 MIA1321
  保留节点；
- `gc2093.c` 接受该板固定 37.125MHz 时钟，同时保留 27MHz 模式表；
- `sysdrv/drv_ko/Makefile` 会将内核产出的 `gc2093.ko` 放入驱动包，`insmod_ko.sh` 会开机加载；
- `luckfox-pico-zero-gc2093-oem-pre.sh` 在生成 `oem.img` 前校验驱动、RKIPC 配置和 IQ，
  并为 DTS 所需的 `gc2093_LUCKFOX-GC2093_30IRC-F20.json` 建立软链接。

复用边界：相同 Pico Zero 硬件、同款 GC2093 模组和相同 MIPI 接线可直接使用此 Profile。
若更换摄像头模组、镜头/IR-Cut、排线或 MIPI 走线，仍须重新确认 IQ、lane 配置与板端
CSI 误码；不能只因镜像可启动就跳过真板验收。

注意：当前 macOS 工作目录位于大小写不敏感文件系统。Linux 内核同时包含
`xt_HL.c` / `xt_hl.c` 等仅大小写不同的文件，在这里会发生冲突，看起来像“缺失源码”，
不能在此目录全量编译。应在 Linux ext4（或 macOS 大小写敏感 APFS 卷）重新克隆 SDK，
再迁入本节列出的 GC2093 Profile 后构建完整 `oem.img/update.img`。
