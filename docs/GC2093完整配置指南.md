## 三、步骤一：编译 gc2093.ko

官方固件 250802 不包含 gc2093.ko，需要手动编译。

### 3.1 关键约束

- 必须与板上 kernel 完全相同的 vermagic：`5.10.160 mod_unload ARMv7 thumb2 p2v8`
- 使用 `arm-linux-gnueabihf-gcc`（apt Ubuntu），**不需要** Rockchip 私有 toolchain
- 板上无 `CONFIG_MODVERSIONS`，无需 CRC 符号校验

### 3.2 Dockerfile

```dockerfile
FROM ubuntu:22.04
ENV DEBIAN_FRONTEND=noninteractive
RUN apt-get update && apt-get install -y \
    gcc-arm-linux-gnueabihf \
    bc flex bison libssl-dev libelf-dev \
    make git wget python3 python-is-python3 cpio rsync \
    && rm -rf /var/lib/apt/lists/*
WORKDIR /build
```

> ⚠️ **必须**加 `python-is-python3`：内核构建脚本调用 `/usr/bin/env python`，Ubuntu 22.04 默认只有 `python3`，缺少此包会导致构建失败。

### 3.3 编译脚本

```bash
#!/bin/bash
set -e

# [1] git sparse-checkout，只拉取内核（节省流量）
mkdir -p /build/kernel-src && cd /build/kernel-src
git init
git remote add origin https://github.com/LuckfoxTECH/luckfox-pico.git
git config core.sparseCheckout true
echo "sysdrv/source/kernel" > .git/info/sparse-checkout
git fetch --depth=1 --filter=blob:none origin main
git checkout main

KDIR=/build/kernel-src/sysdrv/source/kernel

# [2] 配置内核
cd $KDIR
make ARCH=arm luckfox_rv1106_linux_defconfig
./scripts/config --set-val CONFIG_VIDEO_GC2093 m
make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- modules_prepare -j$(nproc)

# [3] 编译 gc2093.ko
make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- \
    M=drivers/media/i2c \
    CONFIG_VIDEO_GC2093=m \
    modules -j$(nproc)

cp $KDIR/drivers/media/i2c/gc2093.ko /output/gc2093.ko
strings /output/gc2093.ko | grep vermagic
```

### 3.4 执行编译

```bash
mkdir -p /tmp/gc2093-output
docker run --rm \
  -v /tmp/gc2093-build/compile.sh:/build/compile.sh \
  -v /tmp/gc2093-output:/output \
  rv1106-builder:latest \
  bash /build/compile.sh
```

### 3.5 验证编译结果

```bash
strings /tmp/gc2093-output/gc2093.ko | grep vermagic
# 期望输出：vermagic=5.10.160 mod_unload ARMv7 thumb2 p2v8
```

## 四、步骤二：准备 IQ 文件

### 4.1 IQ 文件命名规则（关键！）

rkipc 在初始化时会根据 DTS 中的摄像头模组信息拼接 IQ 文件名：

```
{sensor}_{rockchip,camera-module-name}_{rockchip,camera-module-lens-name}.json
```

本项目 DTS 中配置为：
- `rockchip,camera-module-name = "LUCKFOX-GC2093"`
- `rockchip,camera-module-lens-name = "30IRC-F20"`

因此 rkipc 期望的文件名为：
```
gc2093_LUCKFOX-GC2093_30IRC-F20.json
```

### 4.2 从 SDK 提取 IQ 文件

SDK 中实际提供的文件名为 `gc2093_SIDA209300461_60IRC_F20.json`，需要复制并改名：

```bash
# 从 SDK 的 isp_iqfiles 目录提取
cp luckfox-pico/media/isp/.../isp_iqfiles/gc2093_SIDA209300461_60IRC_F20.json \
   /tmp/gc2093-output/
```

### 4.3 部署 IQ 文件到板

```bash
# 部署原始文件
adb push /tmp/gc2093-output/gc2093_SIDA209300461_60IRC_F20.json \
         /oem/usr/share/iqfiles/

# 创建 rkipc 期望命名的文件（与 DTS 中模组名对应）
adb shell "cp /oem/usr/share/iqfiles/gc2093_SIDA209300461_60IRC_F20.json \
              /oem/usr/share/iqfiles/gc2093_LUCKFOX-GC2093_30IRC-F20.json"
```

## 六、步骤四：修改 DTB（关键步骤）

### 6.1 背景：Luckfox boot.img 是 FIT 格式

Luckfox 250802 固件的 `boot.img` 不是普通的 Rockchip 分区格式，而是 **FIT image（Flattened Image Tree）** 格式。

FIT image 结构：
```
offset 0x000:     FIT header DTB（1536 bytes）— 描述各组件位置、大小、SHA256 hash、RSA 签名
offset 0x800:     FDT blob（39020 bytes）— 设备树主体
offset 0xa200:    kernel image（3.2MB）
offset 0x321200:  resource.img（含 rk-kernel.dtb 副本）
offset 0x321a00:  rk-kernel.dtb 数据（39020 bytes）
```

**直接替换 DTB 数据后不生效的根本原因**：FIT header 中存储了 DTB 数据的 SHA256 hash，U-Boot 加载时验证 hash，不匹配则回退到旧的设备树（或内核 appended DTB）。

### 6.2 提取并修改 DTS

```bash
# 1. 提取 FDT blob（offset 0x800，大小 39020 字节）
dd if=boot.img bs=1 skip=$((0x800)) count=39020 of=/tmp/rk-kernel.dtb

# 2. 反编译为 DTS
dtc -I dtb -O dts -o /tmp/rk-kernel.dts /tmp/rk-kernel.dtb

# 3. 编辑 DTS：配置 gc2093@37 节点
```

### 6.3 gc2093 DTS 节点配置

将 DTS 中原摄像头节点替换为以下内容（phandle 引用值保持与原节点相同）：

```dts
gc2093@37 {
    compatible = "galaxycore,gc2093";
    status = "okay";
    reg = <0x37>;
    clocks = <0x41>;
    clock-names = "xvclk";
    reset-gpios = <0x3e 0x15 0x01>;
    pinctrl-names = "default";
    pinctrl-0 = <0x3f>;
    rockchip,camera-module-index = <0x00>;
    rockchip,camera-module-facing = "back";
    rockchip,camera-module-name = "LUCKFOX-GC2093";
    rockchip,camera-module-lens-name = "30IRC-F20";
    port {
        endpoint {
            remote-endpoint = <0x43>;
            data-lanes = <0x01 0x02>;
            phandle = <0x0d>;
        };
    };
};
```

> **关键**：`data-lanes = <0x01 0x02>` 表示 2-lane MIPI，与 GC2093 硬件连接匹配。

### 6.4 重新编译 DTB

```bash
dtc -I dts -O dtb -o /tmp/rk-kernel-new.dtb /tmp/rk-kernel.dts
```

### 6.5 生成修改后的 boot.img（含 hash 更新）

> **适用条件**：新 DTB ≤ 原始 DTB 大小（39020 bytes）。若用 SDK 重新编译的 DTB（42729 bytes），请参考 [常见问题 Q16](./常见问题FAQ.md#q16-sdk-编译的-dtb-比原始-dtb-大无法直接打入-bootimg) 中的 mkimage 方案。

以下 Python 脚本同时完成：替换两处 DTB + 更新 FIT header 中 fdt hash：

```python
import struct, hashlib

ORIG    = '/path/to/boot.img'
NEW_DTB = '/tmp/rk-kernel-new.dtb'
OUT     = '/tmp/boot-gc2093.img'

with open(ORIG, 'rb') as f:   boot = bytearray(f.read())
with open(NEW_DTB, 'rb') as f: new_dtb = f.read()

ORIG_SIZE = 39020  # 原始 DTB 大小（字节）
padded    = new_dtb + b'\x00' * (ORIG_SIZE - len(new_dtb))
new_sha   = hashlib.sha256(padded).digest()

# 验证并替换两处 DTB
for off in [0x800, 0x321a00]:
    assert struct.unpack_from('>I', boot, off)[0] == 0xd00dfeed, f"DTB magic 不对 @ {off:#x}"
    boot[off:off + ORIG_SIZE] = padded

# 更新 FIT header（开头 1536 字节）中 fdt/hash/value 字段
small           = bytearray(boot[:1536])
fdt_off_struct  = struct.unpack_from('>I', small, 8)[0]
fdt_off_strings = struct.unpack_from('>I', small, 12)[0]

FDT_BEGIN_NODE=1; FDT_END_NODE=2; FDT_PROP=3; FDT_NOP=4; FDT_END=9
pos = fdt_off_struct; path = []

while pos < len(small):
    tag = struct.unpack_from('>I', small, pos)[0]; pos += 4
    if tag == FDT_BEGIN_NODE:
        end = small.index(0, pos)
        path.append(small[pos:end].decode('ascii', 'replace'))
        pos = (end + 4) & ~3
    elif tag == FDT_END_NODE:
        if path: path.pop()
    elif tag == FDT_PROP:
        vlen = struct.unpack_from('>I', small, pos)[0]
        noff = struct.unpack_from('>I', small, pos+4)[0]
        voff = pos + 8; pos += 8
        end_s = small.index(0, fdt_off_strings + noff)
        pname = small[fdt_off_strings+noff:end_s].decode('ascii', 'replace')
        full  = '/'.join(path) + '/' + pname
        # 找到 fdt hash.value（32 字节 SHA256）并更新
        if 'images' in full and 'fdt' in full and 'hash' in full and pname == 'value' and vlen == 32:
            small[voff:voff + 32] = new_sha
            print(f"已更新 hash.value @ {voff:#x}")
        pos = (pos + vlen + 3) & ~3
    elif tag == FDT_NOP: pass
    elif tag == FDT_END: break

boot[:1536] = small
with open(OUT, 'wb') as f: f.write(boot)
print(f"生成: {OUT}（{len(boot)} bytes）")
```

```bash
adb reboot loader
```

**恢复情况（板子无法启动，进入 maskrom）：**
```bash
# 按住 BOOT 键后上电 → 进入 maskrom
~/bin/rkdeveloptool db ~/Downloads/Luckfox_Pico_Zero_EMMC_250802/download.bin
sleep 2
~/bin/rkdeveloptool wl 0x640 /tmp/boot-gc2093.img
~/bin/rkdeveloptool rd
```

> **macOS 上 db 挂起的处理**：`db` 在 maskrom 模式偶尔挂起，通过 `adb reboot loader` 进 loader 模式后运行则稳定。遇到挂起先 `pkill rkdeveloptool` 再重试。

## 十、常见问题

### ❌ 问题：DTB 修改后不生效（compatible 未更新）

**原因**：忘记更新 FIT header 中的 SHA256 hash。

**解决**：用第六节的 Python 脚本重新生成 boot.img，该脚本会自动更新 hash。

---

### ❌ 问题：rkipc 启动后立即崩溃

**日志特征**：
```
[video.c][rkipc_get_jpeg]:RK_MPI_VENC_GetStream timeout a004800e
```

**解决**：在 `rkipc-gc2093-200w.ini` 中设置 `enable_jpeg = 0`。

---

### ❌ 问题：rkipc 找不到 IQ 文件

**日志特征**：
```
XCORE:E:access /oem/usr/share/iqfiles/gc2093_LUCKFOX-GC2093_30IRC-F20.json failed!
```

**解决**：IQ 文件名必须与 DTS 中的 `camera-module-name` + `camera-module-lens-name` 一致：
```bash
adb shell "cp /oem/usr/share/iqfiles/gc2093_SIDA209300461_60IRC_F20.json \
              /oem/usr/share/iqfiles/gc2093_LUCKFOX-GC2093_30IRC-F20.json"
```

---

### ❌ 问题：rkdeveloptool db 在 maskrom 模式下挂起（macOS）

**原因**：macOS USB bulk transfer 在 maskrom 模式（DDR 未初始化状态）下兼容性问题。

**解决**：优先使用 `adb reboot loader` 进入 loader 模式再操作；如必须从 maskrom 恢复，多次重试通常可以成功。

---

### ❌ 问题：RTSP 无法连接（Mac 侧）

**原因**：Mac 上的 VPN 将 172.32.x.x 路由到了隧道接口。

**解决**：使用 ADB 端口转发：
```bash
adb forward tcp:15554 tcp:554
ffplay rtsp://localhost:15554/live/0
```
