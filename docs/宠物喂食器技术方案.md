# 宠物智能可视喂食器 — 完整技术方案

> 版本：v3.0  
> 更新日期：2026-07-04

---

## 一、项目概述

**产品功能：**
- 实时视频推流（远程查看宠物状态）
- 宠物个体识别（区分是哪只宠物）
- 进食状态识别（区分进食中与路过）
- 食物余量检测（纯视觉，无传感器）
- 三种喂食触发（APP手动 / 定时 / 进食意图）
- 断网本地缓存（3天），恢复后自动补发

**硬件平台：**

| 组件 | 型号 | 说明 |
|------|------|------|
| 主控 | Luckfox Pico Zero | RV1106G3，单核A7@1.2GHz，1.0 TOPS NPU |
| 内存 | 256MB DDR3L | 可用约190MB（66MB CMA预留） |
| 存储 | eMMC 8GB | 系统+模型+本地缓存 |
| 摄像头（当前） | IMX415 | 采集配置1080P，不超ISP上限 |
| 摄像头（后续） | GC2093 | 200万像素原生，降低ISP压力 |
| MCU | STM32（或同类） | 控制喂食电机，RV1106通过UART发指令 |
| 传感器 | 无 | 食物余量全部依赖视觉 |

**设备为开模量产，摄像头安装位置每台完全一致，碗ROI坐标开发阶段标定一次，硬编码到配置文件。**

## 二、整体架构

```
IMX415 (1080P@25fps)
  │
  ├─ VI Ch0 ──→ VENC H264 CBR ──→ RTSP推流     ← 推流独立，永不阻塞
  │                  ↑
  │             OSD叠加检测框（非阻塞读取最新推理结果）
  │
  └─ VI Ch1 ──→ VPSS(416×416 RGB888)
                      └─ YOLOv5n推理
                              ├─ 宠物框(最高置信度) ──→ Re-ID ──→ 宠物ID
                              │                           └─ 进食判定 + 喂食触发
                              └─ [每30秒] 碗ROI裁剪(128×128) ──→ MobileNetV2 ──→ 余量等级

RV1106 ──UART──→ STM32 ──→ 喂食电机
RV1106 ──MQTT──→ 服务器 ──→ APP推送
```

## 三、三个推理模型

| | 模型一 | 模型二 | 模型三 |
|--|--------|--------|--------|
| 名称 | YOLOv5n | OSNet-x0.25 (Re-ID) | MobileNetV2 |
| 任务 | 检测宠物位置 | 识别是哪只宠物 | 食物余量分类 |
| 输入尺寸 | 416×416 | 128×128 | 128×128 |
| 输出 | cat/dog bounding box | 宠物ID | full/half/low/empty |
| 运行条件 | 每帧 | YOLO检测到宠物时 | 每30秒 |
| 平均帧率 | ~13fps | ~1~3fps | 可忽略 |
| 模型大小 | ~4MB | ~3MB | ~2MB |
| 训练数据 | COCO预训练，直接用 | APP注册照片，服务器提取特征 | 样机出来后采集，覆盖多光线 |

> 三模型共享 RKNN internal memory，实际新增内存接近 0。

## 五、推理模块

### 5.1 YOLOv5n 模型转换

```bash
# 必须加 --rknpu，移除板端不兼容的后处理层
python export.py --rknpu --weight yolov5n.pt --img 416
```

```python
rknn.config(
    mean_values=[[0, 0, 0]],
    std_values=[[255, 255, 255]],
    target_platform='rv1106',
    quantized_algorithm='normal'
)
```

### 5.2 三模型共享内存

```c
rknn_init(&yolo_ctx,  "yolov5n.rknn",  0, RKNN_FLAG_MEM_ALLOC_OUTSIDE, NULL);
rknn_init(&reid_ctx,  "reid.rknn",     0, RKNN_FLAG_MEM_ALLOC_OUTSIDE, NULL);
rknn_init(&food_ctx,  "mobilenet.rknn",0, RKNN_FLAG_MEM_ALLOC_OUTSIDE, NULL);

// 取三者最大internal size，共享一块
size_t max_size = MAX3(yolo_mem.total_weight_size,
                       reid_mem.total_weight_size,
                       food_mem.total_weight_size);
shared_internal = rknn_create_mem(yolo_ctx, max_size);
// 三个模型共享同一块 internal memory（顺序执行，不会冲突）
rknn_set_internal_mem(yolo_ctx, shared_internal);
rknn_set_internal_mem(reid_ctx, shared_internal);
rknn_set_internal_mem(food_ctx, shared_internal);
```

### 5.3 VPSS帧生命周期（必须先拷贝再释放）

```c
RK_MPI_VPSS_GetChnFrame(GrpID, ChnID, &frame, -1);
memcpy(input_mems[0]->virt_addr,
       RK_MPI_MB_Handle2VirAddr(frame.stVFrame.pMbBlk),
       input_size);
RK_MPI_VPSS_ReleaseChnFrame(GrpID, ChnID, &frame); // 拷完再释放
rknn_run(yolo_ctx, NULL);                            // 推理用拷贝数据
```

## 六、线程模型与并发控制

### 6.1 线程结构

```
主线程          初始化所有模块，启动子线程
推流线程        VI Ch0 → VENC → RTSP（最高优先级）
推理线程        VI Ch1 → VPSS → YOLOv5n → [有宠物] → Re-ID → 写共享结果
余量检测线程    每30秒：读碗ROI → 裁剪 → MobileNetV2 → 写food_level
MQTT线程        上报事件；接收APP下发指令（喂食/更新ROI等）
通知线程        监听food_level / feeding_status → UART发指令给STM32
```

### 6.2 推理线程低优先级

```c
struct sched_param param = { .sched_priority = 0 };
pthread_setschedparam(infer_thread, SCHED_IDLE, &param);
```

### 6.3 并发保护

```c
// 检测结果：推流线程非阻塞读（trylock），拿不到用上一帧
if (pthread_mutex_trylock(&result_mutex) == 0) {
    memcpy(draw_results, shared_results, sizeof(draw_results));
    pthread_mutex_unlock(&result_mutex);
}

// 碗位置：余量检测线程短暂持锁，锁外执行推理
pthread_mutex_lock(&bowl_mutex);
local_bowl = shared_bowl_roi;   // 微秒级拷贝
pthread_mutex_unlock(&bowl_mutex);
run_food_classifier(local_bowl);

// 余量等级：atomic，无锁
atomic_store(&current_food_level, level);
FoodLevel lv = atomic_load(&current_food_level);
```

## 七、业务逻辑层

### 7.1 宠物个体识别

```c
// 多只宠物同时入镜：只识别置信度最高的一只
DetectBox best = get_highest_confidence_pet(yolo_results);
char *pet_id   = reid_identify(best, registered_pets);
```

**注册流程（首次使用）：**
1. APP拍摄宠物照片上传服务器
2. 服务器提取Re-ID特征向量，下发到设备
3. 设备存入 `/userdata/pets.json`

### 7.2 进食判定

```c
// 三重条件：连续5帧 + 与碗ROI重叠>30% + 停留>3秒
if (result.consecutive_frames >= 5 &&
    box_overlap(pet_box, bowl_roi) > 0.3f &&
    near_duration > 3 * FPS) {
    status.eating = 1;
}
```

### 7.3 喂食触发

```c
// 三种触发方式并存
// 定时任务时间表存 /userdata/schedule.json，MQTT下发后写入，断网独立执行
switch (trigger_type) {
    case TRIGGER_APP:       // MQTT收到APP下发指令
    case TRIGGER_SCHEDULE:  // 本地定时任务（读schedule.json）
    case TRIGGER_INTENT:    // 检测到进食意图（停留>3秒+靠近碗ROI）
        uart_send_feed_cmd(pet_id);
        mqtt_report_feeding(pet_id, trigger_type, time(NULL));
        break;
}

// 进食意图：暂定不重复触发，预留冷却扩展接口
static int feeding_cooldown = 0;  // 后续可配置冷却时间
if (trigger_type == TRIGGER_INTENT && !feeding_cooldown)
    handle_feed_trigger(TRIGGER_INTENT, pet_id);
```

### 7.4 食物余量检测

```c
// 每30秒，从固定碗ROI裁剪后送MobileNetV2
void check_food_level() {
    cv::Mat bowl_img;
    vpss_crop(bowl_roi_x, bowl_roi_y, bowl_roi_w, bowl_roi_h,
              128, 128, &bowl_img);
    FoodLevel level = (FoodLevel)run_mobilenet(bowl_img);
    atomic_store(&current_food_level, level);
    mqtt_report_food_level(level);
}
```

### 7.5 断网缓存处理

```c
// 断网时写本地SQLite（最多保留3天）
void handle_mqtt_failure(EventRecord *e) {
    sqlite3_insert(local_db, e);
    purge_records_older_than(3 * 86400); // 超3天自动清理
}

// 恢复联网时批量补发
void on_mqtt_reconnect() {
    sqlite3_flush_to_mqtt(local_db);
}
// 断网期间定时喂食正常执行，不依赖网络
```

### 7.6 MQTT上报事件

| 事件 | 触发条件 | 上报内容 |
|------|---------|---------|
| 食物不足 | 余量 low / empty | 余量等级 |
| 宠物进食开始 | 三重条件满足 | 宠物ID + 时间戳 |
| 进食结束 | 离开超10秒 | 宠物ID + 进食时长 |
| 长时间未进食 | 超12小时无记录 | 告警通知 |
| 喂食执行 | 任一触发方式 | 宠物ID + 来源 + 时间 |

## 十二、开发路径

```
阶段一：推流验证（1~2天）
  └─ 跑官方 luckfox_pico_rkmpi_example，验证1080P@25fps稳定性

阶段二：推理验证（2~3天）
  └─ PC端转换YOLOv5n.rknn
  └─ 板端跑通cat/dog检测

阶段三：双通道集成（3~5天）
  └─ 推流+推理双线程，OSD叠加检测框
  └─ 验证推理不影响推流帧率

阶段四：Re-ID集成（3~5天）
  └─ 使用 OSNet-x0.25，转换为 .rknn（~3MB）
  └─ 实现注册流程（服务器提取特征→下发设备→存/userdata/pets.json）
  └─ 多宠物场景验证

阶段五：余量检测（2~3天，依赖样机）
  └─ 样机出来后采集训练数据（多光线，各余量等级各200~300张）
  └─ 训练MobileNetV2，转换部署

阶段六：业务逻辑+MQTT+UART（3~5天）
  └─ 三种喂食触发逻辑
  └─ MQTT上报/接收，断网缓存
  └─ UART发指令给STM32

阶段七：稳定性验收（1~2天）
  └─ 72小时压测，验证无内存泄漏
  └─ 换GC2093，验证ISP压力降低
```

## 十三、关键风险

| 风险 | 可能性 | 应对 |
|------|--------|------|
| IMX415超ISP上限 | 确定 | VI采集强制1080P |
| MobileNetV2依赖样机采集数据 | 高 | **最大时间节点风险**，样机出来前此模块无法完成 |
| 余量分类受光线影响 | 高 | 覆盖白天/夜晚/侧光多场景训练数据 |
| Re-ID多宠物混淆 | 中 | 注册时要求多角度、光线充足的照片 |
| 进食误触发（路过非进食） | 中 | 三重判定条件（帧数+重叠+停留时长） |
| 长时间运行崩溃 | 中 | 资源严格按顺序释放，72小时压测 |
